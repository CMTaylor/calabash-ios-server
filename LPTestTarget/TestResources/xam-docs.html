Introduction to Xamarin Test Cloud

PDF for Offline Use:
  • Download PDF
Related Links:
  • Cucumber - Making BDD Fun
  • NUnit
  • Xamarin Test Cloud Agent

This guide offers a quick introduction to Xamarin Test Cloud and Automated UI Acceptance Testing. It will discuss the underlying concepts and components behind Xamarin Test Cloud and the frameworks that allow developers to write automated UI tests.

Overview

The most effective way to validate the behavior of an application is by running it and using it. If it behaves as expected, without crashing or returning incorrect results, then the application is probably in good shape and can be released to users without worry that it will cause frustration or be unusable. This process, of testing the UI by actually using an application and interacting with it, is known as UI Acceptance Testing. Traditionally, however, UI Acceptance Testing has been extremely expensive due to the manual nature of it. It has historically required lots of manpower to deploy the application and then use it, verifying that it behaves correctly. Multiply that by each build that needs testing as well as each device that it needs testing on, and the cost can skyrocket exponentially, meaning that effective UI Acceptance Testing has been out of reach for nearly everyone but the most resource-laden organizations.

Xamarin Test Cloud is a cloud-based service that provides an automatable way to do UI Acceptance Testing of mobile applications in the cloud across hundreds of different devices. This enables anyone to ensure that their application performs correctly and efficiently across a variety of devices with minimal effort. Additionally, because it’s cloud-based, the maintenance and procurement efforts are then removed from the test consumer, who can then focus on other efforts.

Automated UI tests can be written using one of two frameworks:

  • Xamarin.UITest – This is a framework that allows tests to be written in C# using the popular NUnit testing library. This framework is well suited to developers that are already skilled with writing their own automated tests.
  • Calabash – This is a framework that allows tests to be written in Ruby using Cucumber. Calabash tests are well suited to Behavior Driven Development – a methodology that focuses on creating executable specifications. These are tests that can be written in the language of the business by anyone following grammatical rules imposed by Cucumber.

Regardless of which framework is used to write the tests, these automated tests will run the application's user interface and validate that the Behavior is correct.

Once a piece of functionality and the tests have been created locally, the mobile application and tests are uploaded to Test Cloud, which will install the app and run the tests on hundreds of physical devices. Once all the tests have completed, Test Cloud will send out a notification with the test results.

Whether you are using Calabash or Xamarin.UITest, the application and tests are submitted to Xamarin Test Cloud at a command prompt, a process often referred to as a test run. The submission can be performed manually by a user or as part of a Continuous Integration (CI) workflow that automatically submits new application builds to Xamarin Test Cloud when code changes are checked in:

We shall see how to write these tests later, but first it is important to understand a little bit about the underlying architecture that runs the tests. Let's discuss the Xamarin Test Cloud Agent and the role it plays in automated UI testing.

The Anatomy of the Test Cloud Framework

Calabash and Xamarin.UITest are frameworks for testing – all alone they do not interact with a mobile application. They require some sort of automation library that will run the tests on the device. The Xamarin Test Cloud Agent provides a bridge that allows these tests to run on Android and iOS devices:

The Xamarin Test Cloud Agent is a separate application that is installed by Test Cloud alongside the application. The Xamarin Test Cloud Agent is further split into two pieces, because while the tests run on the device, they are actually initiated and queried externally – either from the command line when run locally, or via Xamarin Test Cloud when run there.

In order to facilitate this, Calabash uses a client-server model in which the server portion is a lightweight web server that runs on the device and listens for a client to connect and initiate the tests. It then communicates via JSON over HTTP.

Therefore, the actual tests run in a host, and they direct actions that are specified in the tests to execute on the device.

Xamarin Test Cloud Agent on iOS

On iOS, the server is built into the app to be tested via a NuGet Package:

As we’ll see in the Introduction to Calabash guide, the Xamarin Test Cloud Agent should only be included in Debug builds of the application, and when building in the Release Configuration, it’s stripped out with the Xamarin.iOS Linker.

Xamarin Test Cloud Agent on Android

In Android, Test Cloud follows the same client-server model, but instead of the server portion running inside your application, it actually runs as a separate application, which has permission to instrument your application. This is possible, because Test Cloud signs both application packages with the same key, so they are allowed to bypass the security sandbox that would typically exist:

The Xamarin Test Cloud Agent is only for iOS projects. It is not necessary for Android projects as the Test Cloud Agent is installed as a separate application.

Running Tests in Xamarin Test Cloud

Running Tests in Xamarin Test Cloud is conceptually similar to running tests locally, except that Xamarin Test Cloud hosts the tests and will execute them on selected devices:

To facilitate this, Calabash tests are uploaded along with the app when it’s pushed to Xamarin Test Cloud for testing. Xamarin Test Cloud then invokes them over HTTP, just as if they were running locally.

Summary

In this guide, we took a quick tour of Test Cloud, introducing the concept of Automated UI Acceptance Testing. Then, we saw how this is possible through tests written in C# using Xamarin.UITest or in Ruby using Calabash. We also learned that App Explorer is handy for quickly seeing how your application will look across major issues.
